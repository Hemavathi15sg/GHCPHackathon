## Section: Naming

### Rule: Variable and function naming case
**Rule:** Use camelCase for all variable names and local function names. Use PascalCase for class names. Use UPPER_SNAKE_CASE for constants. Do not use generic names such as `data`, `temp`, `obj`, `x`, or `thing`.
**Rationale:** Consistent casing makes code scannable. Generic names create ambiguity during maintenance and review.
**Good example:**
```java
String userEmail = request.getEmail();
boolean isActive = user.isEnabled();
static final int MAX_RETRY_COUNT = 3;
```
**Bad example:**
```java
String data = request.getEmail();
boolean active = user.isEnabled();
static final int maxRetryCount = 3;
```
**How to verify:** Search for single-letter variables, `temp`, `data`, `obj`; verify method/variable names use camelCase and class names PascalCase.

## Section: Error Handling

### Rule: Catch specificity
**Rule:** Always catch the most specific exception type available. Never use a bare `catch (Exception e)` unless genuinely handling all failure modes. Never swallow exceptions with an empty catch block.
**Rationale:** Catching broad exceptions hides bugs and makes debugging difficult. Empty catch blocks silently discard failure information.
**Good example:**
```java
try {
    result = parse(input);
} catch (ParseException e) {
    log.error("Parse failed for input: {}", input, e);
    throw new ServiceException("Invalid input format", e);
}
```
**Bad example:**
```java
try {
    result = parse(input);
} catch (Exception e) {
    // nothing
}
```
**How to verify:** Look for `catch (Exception e)` with empty or log-only body. Flag any catch block that does not re-throw or take corrective action.

## Section: Logging

### Rule: Structured logging only
**Rule:** Use the structured JSON logger exclusively. Never use `System.out.println`, `print()`, or raw string concatenation in log calls. Always include `traceId` / correlation ID. Never log passwords, tokens, or PII.
**Rationale:** Structured logs are machine-parseable for alerting and tracing; raw println output is lost in production log aggregators.
**Good example:**
```java
log.info("User registered: userId={}, traceId={}", userId, traceId);
```
**Bad example:**
```java
System.out.println("User registered: " + email + " password=" + password);
```
**How to verify:** Search for `System.out`, `println`, `print(`. Check log lines that reference `password`, `token`, `secret` — these must not appear.

## Section: Security

### Rule: No hardcoded secrets
**Rule:** Never hardcode secrets, API keys, passwords, or tokens in source code or configuration files committed to version control. Use environment variables or a secrets manager.
**Rationale:** Hardcoded secrets are trivially discoverable in source history and scanning tools; they cannot be rotated without a code change.
**Good example:**
```java
String apiKey = System.getenv("PAYMENT_API_KEY");
```
**Bad example:**
```java
private static final String API_KEY = "sk-prod-abc123";
```
**How to verify:** Search for string literals assigned to variables named `key`, `secret`, `password`, `token`. Flag any non-empty string literal in a `final` field with those names.

### Rule: Parameterized queries
**Rule:** Always use parameterized queries or ORM methods. Never concatenate user input into SQL strings.
**Rationale:** SQL injection is OWASP Top 10. String concatenation in queries allows attackers to modify query structure.
**Good example:**
```java
repo.findByEmail(email); // ORM
```
**Bad example:**
```java
"SELECT * FROM users WHERE email = '" + email + "'"
```
**How to verify:** Search for string concatenation (`+`) adjacent to SQL keywords (`SELECT`, `WHERE`, `INSERT`).

## Section: Testing

### Rule: Coverage threshold
**Rule:** Maintain ≥80% line coverage, enforced in CI. All public service methods must have unit tests with mocked external dependencies. Test names must follow `methodName_scenario_expectedResult`.
**Rationale:** Low coverage allows regressions to ship silently. Named tests communicate intent and make failures self-documenting.
**Good example:**
```java
@Test
void register_duplicateEmail_throwsDuplicateEmailException() { ... }
```
**Bad example:**
```java
@Test
void test1() { ... }
```
**How to verify:** Check CI coverage gate. Review test class for method coverage. Verify test names follow the convention.

## Section: Performance

### Rule: No DB queries in loops
**Rule:** Never execute database queries inside loops. Use batch fetch, JOIN, or Spring Data `findAllById` alternatives.
**Rationale:** N+1 query patterns cause exponential DB load and have caused production incidents.
**Good example:**
```java
List<User> users = userRepository.findAllById(ids);
```
**Bad example:**
```java
for (String id : ids) {
    User u = userRepository.findById(id).orElseThrow(); // N+1
}
```
**How to verify:** Look for repository or JDBC calls inside `for`/`while` loops.
